# GEO / AI-Discoverability — Design

**Date:** 2026-06-13
**Status:** Approved (pending spec review)
**Site:** cueism (EmDash + Astro + Cloudflare Workers)

## Goal

Make cueism's product, brand, and article data machine-readable so AI answer
engines (ChatGPT, Perplexity, Google AI Overviews) and classic search can parse
and cite the site accurately. This is "GEO/AEO" work focused on the tactics that
actually move the needle in 2026: structured data and clean machine-readable
metadata — not `llms.txt` hype.

## Context / What Already Exists

Verified against the live site (`cueism.0-b50.workers.dev`):

- **`robots.txt`** — auto-generated by EmDash. Already allows all crawlers
  (`User-agent: * / Allow: /`); only `/_emdash/` is blocked. AI bots are not
  blocked at the robots layer. **No work needed.**
- **`sitemap.xml`** — auto-generated by EmDash. **No work needed.**
- **JSON-LD structured data** — **none on any page** (homepage, posts,
  products). This is the primary gap.
- **OpenGraph** — present but buggy on product pages: `og:type` is `website`
  (should be `product`) and `og:description` contains raw escaped HTML
  (`&lt;h2&gt;...`). To be fixed.

EmDash exposes a JSON-LD pipeline in `emdash/page`, notably
`safeJsonLdSerialize(value)` for safely embedding graphs in a
`<script type="application/ld+json">` tag. We use that helper rather than
hand-rolling serialization (it escapes `</script>`, `<!--`, U+2028/U+2029).

## Non-Goals (Out of Scope)

- `robots.txt` and `sitemap.xml` — owned by EmDash; do not duplicate.
- Cloudflare AI-crawler / Pay-Per-Crawl dashboard settings — operational toggle
  the owner verifies in the Cloudflare dashboard, not code. (Cloudflare blocks
  AI crawlers by default for sites on its network; owner must confirm cueism
  allows the desired AI bots.)
- `FAQPage` / `Review` / `AggregateRating` schema — no FAQ or review data
  exists yet. Revisit when that content exists.
- `llms.txt` is included but explicitly treated as low-value (no major AI
  provider consumes it as of Q1 2026); it is cheap insurance, not a priority.

## Approach

A single pure-function module builds JSON-LD graphs; templates call it and
render one script tag via `safeJsonLdSerialize`. All URLs derive from the
request origin (`Astro.url.origin`) so output is correct on both
`workers.dev` (now) and `cueism.com` (after the WordPress cutover) — nothing
hardcoded.

## Components

### 1. `src/utils/structured-data.ts` (new)

Pure, independently testable functions:

- `stripToPlainText(html, maxLen)` — strip tags + collapse whitespace + truncate.
  Shared by JSON-LD descriptions and the OG-description fix.
- `buildOrganization(origin)` → `Organization` graph: name `"Cueism"`, `url`
  (origin), `logo` (origin + `/favicon.ico`), `email` `0@cueism.com`,
  `areaServed` / country `AU`, and `sameAs: []` (placeholder array so social
  profiles drop in later without a structural change).
- `buildProduct(product, origin)` → `Product` graph with nested offer data:
  `name`, `description` (plain text), `image`, `brand` (→ Organization name),
  and offers using `priceCurrency: "AUD"`. Single price → `Offer`; variant
  price spread → `AggregateOffer` with `lowPrice`/`highPrice`. `availability`
  (`https://schema.org/InStock` | `OutOfStock`) computed from the same
  stock/`trackStock` logic the checkout uses (`utils/products.ts`).
- `buildBlogPosting(post, origin)` → `BlogPosting` graph: `headline`,
  `description`, `image`, `datePublished` (from `getPostDate`), `author` and
  `publisher` (→ Organization).

### 2. `src/pages/shop/[slug].astro`

Render `Product` JSON-LD for the entry. Pass a clean plain-text `description`
and `ogType="product"` to `Base`.

### 3. `src/pages/posts/[slug].astro`

Render `BlogPosting` JSON-LD for the entry.

### 4. `src/layouts/Base.astro`

Always render the `Organization` graph (sitewide brand identity). Extend
`Props` with optional `ogType` (default `"website"`) so product/article pages
can override `og:type`; ensure the clean `description` flows into both the
meta description and OG description.

### 5. `src/pages/llms.txt.ts` (new)

Dynamic route returning `text/plain` markdown: a short site summary plus a
linked list of products and recent posts pulled from the CMS. Low effort,
honest scope.

## Data Flow

```
request → template loads CMS entry (existing getEmDashEntry/Collection)
        → structured-data.ts builds graph from entry.data + Astro.url.origin
        → safeJsonLdSerialize(graph) → <script type="application/ld+json">
Base.astro → buildOrganization(origin) on every page (sitewide)
```

No new data sources, no new bindings, no DB changes. Reuses existing
`utils/products.ts` and `utils/post-format.ts` helpers.

## Error Handling

- Builders return `null` / omit keys when source fields are absent;
  `cleanJsonLd`-style null-omission so validators stay happy.
- A failed/empty entry already redirects to 404 before schema runs.
- `safeJsonLdSerialize` handles injection-safe embedding.
